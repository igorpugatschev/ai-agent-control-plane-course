# Модуль 1. Основа control plane

Модуль переводит работу с LLM из режима разового диалога в управляемый инженерный процесс. Сквозной пример во всех трех уроках - неоднозначный запрос на изменение учебного приложения задач:

> Нужно улучшить задачи: добавить нормальные статусы, сделать создание удобнее и закрыть старые проблемы. Желательно, чтобы всё работало как в API.

К концу модуля студент не реализует этот запрос, а превращает его в безопасное и проверяемое описание работы. Это намеренное ограничение: сначала согласуются цель, границы, владельцы решений и доказательства, затем начинается изменение кода.

## Маршрут

1. [От LLM-чата к управляемому процессу](lesson-01-chat-to-control-plane.md) - отличить LLM, prompt, agent, workflow и control plane. Артефакт: `artifacts/module-01/change-request-comparison.md`.
2. [Границы задачи и ответственность](lesson-02-task-boundaries.md) - превратить неоднозначный запрос в контракт с разрешениями и stop-условиями. Артефакт: `artifacts/module-01/task-boundary-contract.md`.
3. [Control-plane blueprint v1](lesson-03-blueprint-v1.md) - собрать контекст, роли, workflow, gates, evidence и риски. Артефакт: `artifacts/module-01/control-plane-blueprint.md`.
4. [Checkpoint 1](checkpoint.md) - самостоятельно построить ограниченный blueprint по исходному сценарию.

Каждый следующий шаг использует решения предыдущего. Таблица из урока 1 помогает выбрать нужный уровень управления; контракт из урока 2 становится входом для blueprint; checkpoint проверяет итоговый `control-plane-blueprint.md` как единый накопленный артефакт.

## Режимы прохождения

Обязательный режим работает без API-ключа: студент читает подготовленный ответ агента, проверяет его по критериям и создает Markdown-артефакт локально. Для желающих в каждом уроке есть необязательный prompt для живого агента. Ответ живого агента не считается доказательством сам по себе и проверяется по тем же локальным критериям.

## Готовность модуля

Модуль завершен, если:

- в `artifacts/module-01/control-plane-blueprint.md` есть измеримая цель, включенный и исключенный scope;
- неоднозначные решения не выданы за подтвержденные требования;
- указаны человеческий владелец приемки, разрешенные действия и отдельное подтверждение для необратимых действий;
- workflow содержит проверяемый gate, evidence и условие остановки;
- checkpoint пройден без критических дефектов.
