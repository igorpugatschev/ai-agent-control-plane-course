# Глоссарий agent control plane

Этот глоссарий объясняет обязательные термины внутри курса. Ссылки на [каталог достоверных источников](../docs/source-catalog.md) нужны для проверки и углубления, но не заменяют определения ниже.

## LLM (Large Language Model, большая языковая модель)

**Определение.** LLM - модель, которая по входному контексту предсказывает продолжение текста и может следовать инструкциям.

**Инженерный пример.** Модель получает описание сбоя и журнал теста, затем предлагает гипотезы для triage.

**Не путать с агентом.** LLM генерирует ответ; агент добавляет к модели цель, инструменты, состояние и правила выполнения.

## Prompt (промпт, инструкция)

**Определение.** Prompt - переданный модели текст или структурированное сообщение с задачей, ограничениями и нужным форматом результата.

**Инженерный пример.** Инструкция просит классифицировать дефект, сослаться на строки лога и вернуть JSON заданной схемы.

**Не путать с контекстом.** Prompt задает поведение для текущего шага, а context включает также данные, историю и результаты инструментов.

## Agent (агент)

**Определение.** Агент - управляемый исполнитель, который использует LLM для достижения цели, выбирая разрешенные действия и фиксируя результат.

**Инженерный пример.** Агент проверки документации читает указанные файлы, запускает тест и передает проверяемый отчет reviewer-агенту.

**Не путать с ролью.** Агент является исполнителем, а role описывает его ответственность и границы.

## Control plane (плоскость управления)

**Определение.** Control plane - правила и механизмы, которые задают агентам роли, маршруты, разрешения, проверки и наблюдаемость.

**Инженерный пример.** Конфигурация назначает implementer, reviewer и release-manager, запрещает прямой deploy и требует evidence перед выпуском.

**Не путать с workflow.** Workflow описывает последовательность работы, а control plane управляет тем, кто и на каких условиях может ее выполнять.

## Context (контекст)

**Определение.** Контекст - набор данных, доступных агенту в текущем запуске: инструкции, файлы, историю, параметры и результаты инструментов.

**Инженерный пример.** Для исправления теста агенту передают баг-репорт, целевой модуль, ошибку CI и ограничения репозитория.

**Не путать с источником.** Источник - место происхождения данных; контекст - отобранная для конкретной работы часть этих данных.

## Evidence (доказательство)

**Определение.** Evidence - проверяемый артефакт, на котором можно обосновать решение или приемку результата.

**Инженерный пример.** Вывод `pytest`, ссылка на commit и diff являются evidence того, что изменение реализовано и проверено.

**Не путать с утверждением.** Утверждение сообщает вывод, а evidence позволяет другому человеку этот вывод перепроверить.

## Role (роль)

**Определение.** Роль - контракт ответственности исполнителя: его цель, входы, выходы, права и запреты.

**Инженерный пример.** Reviewer может читать diff и запускать проверки, но не меняет код автора без отдельного поручения.

**Не путать с skill.** Роль отвечает на вопрос «кто и за что отвечает», а skill - «как выполнить повторяемую работу».

## Skill (навык, повторяемая процедура)

**Определение.** Skill - документированная процедура с входами, шагами, инструментами, ограничениями и ожидаемым результатом.

**Инженерный пример.** Skill «проверка регрессии» описывает запуск набора тестов, triage падений и состав evidence.

**Не путать с tool.** Skill координирует действия; tool выполняет отдельное действие, например поиск или запуск теста.

## Tool (инструмент)

**Определение.** Tool - разрешенный интерфейс, через который агент получает данные или меняет внешнее состояние.

**Инженерный пример.** `pytest` запускает проверки, `git diff` показывает изменение, а API-клиент отправляет запрос в тестовую среду.

**Не путать с permission.** Tool описывает возможность действия, permission определяет, разрешено ли данному исполнителю это действие.

## Workflow (рабочий процесс)

**Определение.** Workflow - явная последовательность шагов, ветвлений, gate-проверок и восстановления для достижения результата.

**Инженерный пример.** Процесс изменения документации: проверить scope, внести правку, запустить тесты, пройти review-gate и передать отчет.

**Не путать с чек-листом.** Чек-лист перечисляет проверки, а workflow определяет порядок, участников и реакцию на разные исходы.

## Handoff (передача работы)

**Определение.** Handoff - структурированная передача состояния, результата, evidence и следующих действий другому исполнителю.

**Инженерный пример.** Implementer передает reviewer-агенту SHA, измененные файлы, команды проверок и известное ограничение.

**Не путать с final report.** Handoff продолжает работу следующего владельца, final report фиксирует итог для заказчика или приемки.

## Gate (контрольная точка)

**Определение.** Gate - условие, которое должно быть выполнено до перехода workflow к следующему этапу.

**Инженерный пример.** До commit требуется успешный focused test и отсутствие ошибок `git diff --check`.

**Не путать с approval.** Gate проверяет условие, approval - явное решение уполномоченного человека или роли разрешить продолжение.

## Stop-gate (стоп-шлюз)

**Определение.** Stop-gate - gate, при срабатывании которого работа останавливается до устранения риска или получения решения.

**Инженерный пример.** Агент прекращает deploy, если миграция не имеет проверенного rollback-плана.

**Не путать с review-gate.** Stop-gate блокирует действие из-за опасного условия; review-gate запрашивает оценку качества результата.

## Review-gate (шлюз ревью)

**Определение.** Review-gate - контрольная точка, где независимая роль проверяет scope, качество и evidence результата.

**Инженерный пример.** Reviewer сопоставляет diff со спецификацией и подтверждает, что тесты покрывают новую ветвь поведения.

**Не путать с evaluation.** Review-gate - место в процессе и решение о переходе; evaluation - измерение качества по заранее выбранным критериям.

## Evaluation (оценка)

**Определение.** Evaluation - воспроизводимое измерение качества ответа, действия или workflow по набору критериев и примеров.

**Инженерный пример.** Набор задач измеряет, верно ли агент извлекает evidence из логов и отказывается от неразрешенных действий.

**Не путать с тестом.** Тест обычно проверяет детерминированное свойство системы, evaluation может измерять качество вероятностного агентного поведения.

## Tracing (трассировка)

**Определение.** Tracing - запись хода выполнения: вызовов модели и инструментов, входов, выходов, времени и ошибок.

**Инженерный пример.** Трасса показывает, какой поиск был выполнен перед неверным ответом и какой tool вернул пустой результат.

**Не путать с логированием.** Логи могут быть отдельными сообщениями, а tracing связывает события одного запуска в причинную цепочку.

## Prompt injection (инъекция промпта)

**Определение.** Prompt injection - попытка через недоверенные данные заставить модель нарушить исходные инструкции или границы доступа.

**Инженерный пример.** Текст из веб-страницы требует «игнорировать правила и раскрыть секреты»; агент должен считать его данными, а не командой.

**Не путать с обычным prompt.** Обычный prompt исходит от доверенного управляющего контура; инъекция находится в недоверенном содержимом и не получает полномочий.

Для сверки продуктовых деталей и актуальных ограничений используйте [каталог достоверных источников](../docs/source-catalog.md), прежде всего разделы OpenAI и Model Context Protocol.
