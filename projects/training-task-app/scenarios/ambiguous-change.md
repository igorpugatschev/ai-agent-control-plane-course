# Неоднозначный запрос на изменение

## Входящий запрос

> Нужно улучшить задачи: добавить нормальные статусы, сделать создание удобнее и закрыть старые проблемы. Желательно, чтобы всё работало как в API.

## Почему запрос нельзя сразу выполнять

В запросе не определены допустимые статусы, правила названий, формат API, границы «старых проблем», владелец решения и критерии готовности. Нельзя проверить результат только по этой формулировке.

## Вопросы для уточнения

1. Какие операции входят в изменение: создание, список, завершение, удаление или другие?
2. Какие статусы нужны и какое действие переводит задачу между ними?
3. Должны ли пробелы по краям названия удаляться?
4. Что считать дубликатом: сравнивать названия с учётом регистра или без него?
5. Какой контракт и какие точные ошибки должны быть у API?
6. Кто принимает результат и каким тестом подтверждается готовность?

## Зафиксированная граница стенда

Для учебного примера принято минимальное решение: создать, получить список и завершить задачу; статусы `open` и `done`; идентификаторы начинаются с `1`; пустые названия и активные дубликаты отклоняются; данные живут только в памяти одного `TaskService`. Остальные пожелания в этот стенд не входят.
